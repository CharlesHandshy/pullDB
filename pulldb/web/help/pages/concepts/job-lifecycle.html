Job Lifecycle

Understanding how restore jobs move through the system from submission to completion. This knowledge helps you monitor progress, diagnose issues, and understand what's happening behind the scenes.

Overview

When you submit a restore job, it goes through a series of states and phases. States represent the overall job status (queued, running, complete), while phases describe what the worker is doing during the running state.

Job States

Active States

queued ⏳

Job is waiting for a worker to pick it up. Jobs are processed in FIFO order.

Duration: Typically seconds
running ⚙️

A worker is actively processing this job. See phases for what happens during this state.

Duration: 2-15 minutes typical
canceling 🛑

User requested cancellation. Worker is cleaning up resources before marking canceled.

Duration: Seconds to 1 minute
deployed 🚀

Database is live and ready for use. This is the primary working state.

Duration: Until expiration or user action

Terminal States

No further transitions possible. The job record is retained for history.

complete ✅

User clicked "Mark Done" in the Web UI. The job moves to History and the database is queued for deletion (deleting → deleted). The database is not immediately dropped — deletion is processed asynchronously.

Next step: Database queued for deletion
failed ❌

Something went wrong. Check events for details and see failure modes.

Next step: Review error, then resubmit from History view
canceled 🚫

User canceled the job. Resources have been cleaned up.

Next step: Submit a new job if needed
deleted 🪦

Database has been dropped. Job record retained for history.

Final: No further actions possible

Cleanup States

Intermediate states that eventually transition to deleting → deleted. These are not final — the job still has pending transitions.

expired ⏰

Retention period exceeded. After the cleanup grace period elapses, the database is automatically dropped (expired → deleting → deleted).

Prevention: Lock database or extend retention
superseded 🔄

A newer restore to the same target replaced this database. The target was overwritten in place by the new restore; db_dropped_at is stamped immediately. The record is retained in history.

Automatic: Triggered when a new restore deploys to the same target
deleting 🗑️

Database DROP is in progress. Brief transitional state — transitions to deleted when the drop completes.

Duration: Seconds

Processing Phases

While a job is in the running state, the worker progresses through several phases. Each phase performs a specific task in the restore pipeline.

flowchart LR subgraph running A[discovery] --> B[download] B --> C[extraction] C --> D[myloader] D --> E[post_sql] E --> F[atomic_rename] end F --> G((deployed))
🔍

discovery

Searches S3 for matching backups and selects the best one based on date and customer name.

Typical time: 1-5s Depends on: S3 listing speed, number of backups
📥

download

Downloads the compressed backup from S3 to local storage. Progress is tracked and reported.

Typical time: 30s - 5min Depends on: Backup size, network speed
📦

extraction

Decompresses the backup archive. Output size is typically 2-3x the compressed size.

Typical time: 15s - 2min Depends on: Backup size, CPU speed
🔄

myloader

Runs myloader to load the data into a staging database. This is usually the longest phase.

Typical time: 1-10min Depends on: Data size, DB performance, indexes

⚠️ Note: Jobs cannot be canceled once this phase begins.

🔧

post_sql

Executes post-restore SQL scripts. These typically anonymize sensitive data (emails, passwords, PII) making the database safe for development.

Typical time: 5-30s Depends on: Script complexity
✨

atomic_rename

Performs the atomic rename from staging to target database. This is instantaneous and ensures zero downtime.

Typical time: <1s Guarantee: Atomic, no partial states

State Transitions

From To Trigger
queued running Worker claims job from queue
running deployed All phases finish successfully
running failed Unrecoverable error in any phase
queued canceled User runs pulldb cancel
running canceling User runs pulldb cancel
canceling canceled Worker finishes cleanup
deployed expired Retention period exceeded (if not locked)
deployed complete User marks "done" in Web UI
deployed superseded User restores same database again
expired deleting Retention cleanup job runs
superseded deleting Automatic cleanup after supersession
complete deleting User requests deletion
deleting deleted Database DROP completes
Terminal states are final
Once a job reaches deleted, failed, or canceled, it cannot be modified. However, failed jobs can be resubmitted from the History view—this creates a new job with the same backup and settings.
Protect your databases
Lock deployed databases to prevent automatic expiration. You can also extend retention up to 12 months.

Database Lifecycle

Once a restore completes, the database enters the deployed state and has its own lifecycle managed by retention policies, user actions, and automatic cleanup.

Retention System

Each deployed database has an expiration time based on retention settings:

  • Default retention: 7 days (configurable by admin)
  • Maximum extension: Up to 12 months
  • Expiration warning: Notification before expiry

Protecting Databases

🔒

Lock

Prevents automatic expiration. Use for databases needed indefinitely.

⏱️

Extend

Add time to retention (1-12 months). Resets expiration clock.

Supersession

When you restore the same database while you already have a deployed version, the old one is automatically marked superseded and queued for deletion. This prevents accumulation of stale databases.

Avoid supersession
Use --custom-target to give databases unique names if you need multiple versions of the same source.

Failure Modes

When a job fails, the error message and phase indicate what went wrong:

Phase: downloading

Causes:

  • S3 credentials expired or invalid
  • Backup file doesn't exist (wrong date?)
  • Network connectivity issues
  • S3 bucket permissions

Resolution: Check your S3 configuration and try again. Verify the backup exists with pulldb list <customer>.

Phase: restoring

Causes:

  • Corrupted backup file
  • MySQL connection issues
  • Insufficient disk space
  • MySQL permissions for staging user

Resolution: Check disk space and MySQL connectivity. Review myloader logs in job events.

Phase: finalizing

Causes:

  • Target database exists and overwrite flag not set

Resolution: Either add overwrite to your command or manually drop the existing database.

Phase: post_sql

Causes:

  • SQL syntax error in post-SQL script
  • Referenced table/column doesn't exist
  • Timeout on large UPDATE statements

Resolution: Review the post-SQL script and check the error message in job events.
