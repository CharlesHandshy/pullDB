Help CLI Reference

CLI Reference

Complete reference for pullDB command-line tools. The pulldb command is for end users to submit and monitor restore jobs. The pulldb-admin command is for system administrators.

Quick Reference

Quick Reference
# Submit a restore
$ pulldb restore <customer>
# Check your last job status
$ pulldb status
# Stream events in real-time
$ pulldb status --rt
# Cancel a job
$ pulldb cancel <job_id>

pulldb restore

Submit a restore job to download a backup from S3 and restore it to your development database.

pulldb restore <customer> [options] User Command

Parameters

Parameter Required Default Description
<customer> Yes* - Customer ID (positional, lowercase letters only)
qatemplate Yes* - Restore QA template instead of customer
dbhost= No localhost Target database host
suffix= No - Suffix for target database (1-3 lowercase letters)
date= No Latest Specific backup date (YYYY-MM-DD)
s3env= No prod S3 environment (staging|prod|both)
overwrite No false Overwrite existing target database

* Either <customer> or qatemplate is required, but not both. Username is auto-detected from your SSH session.

Target Database Naming

Your restored database is named using your user code (first 6 alpha characters of username). All names use lowercase letters only [a-z] — no uppercase, numbers, symbols, or spaces:

<usercode><customer><suffix>

Example: User charles (code: chrles) restoring customer acme with suffix dev:

  • Target database: chrlesacme (or chrlesacmedev with suffix)
  • Staging (temp): chrlesacmeabcdefgh (8-char job ID prefix appended)

Examples

Examples
# Basic restore - latest backup
$ pulldb restore acme
# Restore specific date
$ pulldb restore acme date=2026-01-01
# Restore to remote host
$ pulldb restore acme dbhost=dev-db-01
# Restore with suffix for multiple copies
$ pulldb restore acme suffix=dev
# Overwrite existing database
$ pulldb restore acme overwrite

pulldb status

Check the status of your jobs. Without arguments, shows your last submitted job.

pulldb status [job_id] [options] User Command

Arguments & Options

job_id Optional - Specific job ID (or prefix) to check
--active Optional false Show only active jobs (queued/running)
--history Optional false Show historical jobs (completed/failed/canceled)
--status Optional - Filter by status (queued|running|complete|failed|canceled)
--limit Optional 100 Maximum number of jobs to display
--rt Optional false Stream job events in realtime (auto-exits when job completes)
--json Optional false Output JSON instead of table
--wide Optional false Show additional columns (staging_name)

Output

Status Output

OPERATION: Restore progress
JOB_ID : 7f3a2b1c
CUSTOMER : acme
TARGET : chrlesacme
DB : localhost
USER : chrles
SUBMITTED: 2026-01-03T10:42:15
STARTED : 2026-01-03T10:42:18

Job States

QUEUED RUNNING COMPLETE DEPLOYED FAILED CANCELED

See Job Lifecycle for detailed state descriptions.

pulldb cancel

Cancel a running or queued job. Gracefully stops the job and cleans up resources.

pulldb cancel <job_id> [options] User Command

Options

-f, --force No false Skip confirmation prompt

What Gets Cleaned Up

  • Staging database is dropped (if created)
  • Downloaded backup files are removed
  • S3 download is aborted (if in progress)

Cancellation Window

Jobs can only be canceled before the myloader restore process begins. Once myloader starts loading data into the database, cancellation is blocked to prevent database corruption.

If you attempt to cancel during the loading phase, you'll receive: "Cannot cancel: job has entered loading phase. Myloader cannot be safely interrupted."

Note: Completed jobs cannot be canceled. The database will remain until overwritten by a new restore with overwrite flag.

pulldb search

Search for customers by name pattern. Use wildcards (* or ?) for partial matches.

pulldb search <query> [options] User Command

Parameters

limit=N No 20 Maximum results to show
json No false Output JSON instead of list

Example

Search Customers
$ pulldb search action
Customers matching 'action':
action
actionpest
actiontermite
actionplus
4 customer(s) found.
Use 'pulldb list <customer>' to see available backups.

Tip: Use pulldb list <customer> to see available backups for a specific customer.

Admin Commands

The pulldb-admin command provides system administration capabilities. Requires admin role.

pulldb-admin settings

pulldb-admin settings <action> Admin

Actions

listList all settings with their current values
get <key>Get the value of a specific setting
set <key> <value>Set a setting value in BOTH database and .env
reset <key>Remove a setting from the database
diffShow differences between database and .env file
pullSync settings from database to .env file
pushSync settings from .env file to database
exportExport all current settings

pulldb-admin secrets

pulldb-admin secrets <action> Admin

Manage AWS Secrets Manager credentials for database hosts.

Actions

listList all secrets with the given prefix
get <name>Get a secret's value
set <name>Create or update a secret
delete <name>Delete a secret
rotate-host <host>Safely rotate credentials - atomic MySQL + AWS update with verification
test <name>Test MySQL connectivity using a secret

rotate-host Details

Performs atomic credential rotation for a database host:

  1. Validates current credentials work on MySQL
  2. Verifies user has ALTER USER privilege
  3. Generates new secure password (or uses provided)
  4. Updates MySQL user password (ALTER USER)
  5. Verifies new password works
  6. Updates AWS Secrets Manager
  7. Final round-trip verification

If MySQL succeeds but AWS fails, provides manual fix instructions.

Global Options

--profileAWS profile to use
--regionAWS region
