Help CLI Reference

CLI Reference

Complete reference for pullDB command-line tools. The pulldb command is for end users to submit and monitor restore jobs. The pulldb-admin command is for system administrators.

Quick Reference

Quick Reference
# Submit a restore
$ pulldb restore <customer>
# Check your last job status
$ pulldb status
# Stream events in real-time
$ pulldb status --rt
# Cancel a job
$ pulldb cancel <job_id>

pulldb restore

Submit a restore job to download a backup from S3 and restore it to your development database.

pulldb restore <customer> [options] User Command

Source Selection

Parameter Required Description
<customer> Yes* Customer ID (positional, lowercase letters only)
qatemplate Yes* Restore QA template instead of customer

* Either <customer> or qatemplate is required, but not both.

Target Options

Option Default Description
dbhost= Default host Target database host
suffix= - Suffix for target database (1-3 lowercase letters, pattern: ^[a-z]{1,3}$)
target=<name> - Override target database name (1-51 lowercase letters, pattern: ^[a-z]{1,51}$)
overwrite false Overwrite existing target database
Custom Target + Suffix
target= and suffix= cannot be used together. Include any suffix directly in your custom target name.

Backup Selection

Option Default Description
date= Latest Specific backup date (YYYY-MM-DD)
s3env= prod S3 environment: staging, prod, or both

Admin Options

Option Description
user=<username> Submit job for another user (admin only)

Target Database Naming

Your restored database is named using your user code (first 6 alpha characters of username). All names use lowercase letters only [a-z]:

<usercode><customer>[<suffix>]

Example: User charles (code: chrles) restoring customer acme:

  • Default: chrlesacme
  • With suffix dev: chrlesacmedev
  • Custom target: Whatever you specify (1-51 lowercase letters)

Examples

Examples
# Basic restore - latest backup
$ pulldb restore acme
# Restore QA template
$ pulldb restore qatemplate
# Restore specific date
$ pulldb restore acme date=2026-01-01
# Restore with custom target name
$ pulldb restore acme target=myprojectdev
# Restore with suffix for multiple copies
$ pulldb restore acme suffix=dev
# Overwrite existing database
$ pulldb restore acme overwrite
# Admin: restore on behalf of another user
$ pulldb restore acme user=jsmith

pulldb status

Check the status of your jobs. Without arguments, shows your last submitted job.

pulldb status [job_id] [options] User Command

Arguments & Options

Argument/OptionRequiredDefaultDescription
job_id Optional - Specific job ID (or prefix) to check
--active Optional false Show only active jobs (queued/running)
--history Optional false Show historical jobs (complete/failed/canceled)
--status Optional - Filter by status (queued|running|complete|failed|canceled)
--limit Optional 100 Maximum number of jobs to display
--rt Optional false Stream job events in realtime (auto-exits when job completes)
--json Optional false Output JSON instead of table
--wide Optional false Show additional columns (staging_name)

Output

Job States

Jobs can be in one of 11 possible states:

queued running canceling complete deployed expired superseded failed canceled deleting deleted

pulldb cancel

Cancel a running or queued job. Gracefully stops the job and cleans up resources.

pulldb cancel <job_id> [options] User Command

Options

OptionRequiredDefaultDescription
-f, --force No false Skip confirmation prompt

What Gets Cleaned Up

  • Staging database is dropped (if created)
  • Downloaded backup files are removed
  • S3 download is aborted (if in progress)

Cancellation Behavior

Both queued and running jobs can be canceled. The behavior depends on the current job state:

  • Queued jobs: Canceled immediately.
  • Running jobs: The job transitions to canceling state. The worker stops at its next checkpoint (after the current myloader phase completes), then moves the job to canceled and cleans up resources.

Cancellation is not blocked once myloader starts — it is deferred until the checkpoint. Cleanup (staging database drop, work file removal) happens after the checkpoint is reached.

Note: Jobs in terminal states (complete, deployed, expired, failed) cannot be canceled. The database will remain until overwritten by a new restore with the overwrite flag.

pulldb register

Register a new pullDB account or request access from a new machine. Creates a user account linked to your SSH username.

pulldb register [--password TEXT] User Command

Options

OptionRequiredDefaultDescription
--password No - Password for the account (minimum 8 characters). Prompted if not provided.

Behavior

  • New User: Creates a new account with your system username. Account is disabled until approved by admin.
  • Existing User, New Machine: If you already have an account but run from a new computer, it automatically requests a new API key for this host.

Example: New User

Register New Account

✓ Account created successfully!
Username: charles
User Code: chrles
✓ API credentials saved to ~/.pulldb/credentials

Contact an administrator to enable your account.

Example: Existing User, New Machine

Request Access From New Machine

✓ API key requested successfully!
Username: charles
User Code: chrles
Hostname: my-laptop
✓ API credentials saved to ~/.pulldb/credentials

Pending Approval: New accounts and API keys require admin approval. Contact an admin to: (1) enable your account with pulldb-admin users enable <username>, and (2) approve your API key with pulldb-admin keys approve <key_id>.
One key per machine: Each computer needs its own API key. Simply run pulldb register on any new machine—it will detect you already have an account and request a new key automatically.

pulldb-admin secrets

pulldb-admin secrets <action> Admin

Manage AWS Secrets Manager credentials for database hosts.

Actions

ActionDescription
listList all secrets with the given prefix
get <name>Get a secret's value
set <name>Create or update a secret
delete <name>Delete a secret
rotate-host <host>Safely rotate credentials - atomic MySQL + AWS update with verification
test <name>Test MySQL connectivity using a secret

rotate-host Details

Performs atomic credential rotation for a database host:

  1. Validates current credentials work on MySQL
  2. Verifies user has ALTER USER privilege
  3. Generates new secure password (or uses provided)
  4. Updates MySQL user password (ALTER USER)
  5. Verifies new password works
  6. Updates AWS Secrets Manager
  7. Final round-trip verification

If MySQL succeeds but AWS fails, provides manual fix instructions.

Global Options

OptionDescription
--profileAWS profile to use
--regionAWS region
