API Reference

pullDB provides a RESTful API for programmatic access. The API uses JSON for request and response bodies. Most users will use the CLI, but the API is available for integration into other tools.

Base URL

https://pulldb.yourcompany.com/api/v1

Replace with your actual pullDB server URL.

Authentication

API requests require authentication via API key or session token.

API Key Authentication

Pass your API key in the X-API-Key header:

curl
$ curl -H "X-API-Key: your-api-key" \
https://pulldb.example.com/api/v1/jobs

Submit Job

Create a new restore job.

POST /jobs

Request Body

{
  "user": "charles",
  "customer": "acme",
  "dbhost": "localhost",
  "date": "2026-01-01",
  "overwrite": false
}
Field Type Required Description
user string Yes Username for job ownership
customer string Yes* Customer ID
qatemplate boolean Yes* Restore QA template instead
dbhost string No Target database host (default: localhost)
date string No Backup date (YYYY-MM-DD), defaults to latest
overwrite boolean No Overwrite existing database (default: false)

* Either customer or qatemplate required

Response

{
  "job_id": "7f3a2b1c-85a1-4da2-9c8e-1234567890ab",
  "status": "queued",
  "target_db": "chrlesacme",
  "created_at": "2026-01-02T10:42:15Z"
}

Example

curl
$ curl -X POST \
-H "X-API-Key: your-key" \
-H "Content-Type: application/json" \
-d '{"user":"charles","customer":"acme"}' \
https://pulldb.example.com/api/v1/jobs

Get Job Status

Get the current status of a job.

GET /jobs/{job_id}

Response

{
  "job_id": "7f3a2b1c-85a1-4da2-9c8e-1234567890ab",
  "user": "charles",
  "customer": "acme",
  "status": "running",
  "phase": "restoring",
  "target_db": "chrlesacme",
  "dbhost": "localhost",
  "created_at": "2026-01-02T10:42:15Z",
  "started_at": "2026-01-02T10:42:18Z"
}

List Jobs

List jobs for the authenticated user.

GET /jobs

Query Parameters

status string No Filter by status (queued, running, complete, failed, canceled)
limit integer No Max results (default: 20)
offset integer No Pagination offset

Response

{
  "jobs": [
    {
      "job_id": "7f3a2b1c-85a1-4da2-9c8e-1234567890ab",
      "status": "complete",
      "customer": "acme",
      ...
    }
  ],
  "total": 42,
  "limit": 20,
  "offset": 0
}

Cancel Job

Cancel a queued or running job.

DELETE /jobs/{job_id}

Response

{
  "job_id": "7f3a2b1c-85a1-4da2-9c8e-1234567890ab",
  "status": "canceled",
  "message": "Job canceled successfully"
}
Returns 409 Conflict if job is already in terminal state (complete, failed, canceled), or if the job has entered the loading phase (myloader cannot be safely interrupted).

GET /backups/search

HTTP Status Codes

Code Meaning
200 Success
201 Created (job submitted)
400 Bad Request (invalid parameters)
401 Unauthorized (missing/invalid API key)
403 Forbidden (insufficient permissions)
404 Not Found (job/backup doesn't exist)
409 Conflict (e.g., can't cancel completed job)
500 Internal Server Error

Error Handling

Error responses include a JSON body with error details:

{
  "error": {
    "code": "BACKUP_NOT_FOUND",
    "message": "No backup found for customer 'acme' on date '2026-01-01'",
    "details": {
      "customer": "acme",
      "date": "2026-01-01"
    }
  }
}

Error Codes

INVALID_REQUEST Missing or invalid parameters
USER_NOT_FOUND User doesn't exist
USER_DISABLED User account is disabled
BACKUP_NOT_FOUND Backup doesn't exist
DATABASE_EXISTS Target database exists and overwrite=false
JOB_NOT_FOUND Job ID doesn't exist
