API Reference

pullDB provides a RESTful API for programmatic access. The API uses JSON for request and response bodies. Most users will use the CLI, but the API is available for integration into other tools.

Documentation Scope
This page covers the most commonly used API endpoints for CLI integration. The full API includes 53 endpoints. For complete coverage including admin APIs, see the OpenAPI documentation or use GET /api/openapi.json.

Base URL

http://pulldb-server:8080/api

Replace pulldb-server with your actual pullDB server hostname. Default port is 8080.

Authentication

pullDB uses HMAC-signed requests for secure authentication. All API requests must include signature headers.

HMAC Signed Authentication (CLI/Programmatic)

CLI tools and programmatic clients must sign every request with HMAC-SHA256. Required headers:

  • X-API-Key: Your API key identifier
  • X-Timestamp: ISO 8601 UTC timestamp (e.g., 2026-01-03T15:42:00Z)
  • X-Signature: HMAC-SHA256 signature (hex-encoded)
curl with HMAC signature
$ curl -H "X-API-Key: $PULLDB_API_KEY" \
-H "X-Timestamp: 2026-01-03T15:42:00Z" \
-H "X-Signature: a1b2c3d4e5f6..." \
http://pulldb-server:8080/api/jobs

Signature Computation

The signature is computed over a canonical string:

string_to_sign = f"{METHOD}\n{PATH}\n{TIMESTAMP}\n{BODY_SHA256}"
signature = HMAC-SHA256(secret, string_to_sign)

Where BODY_SHA256 is the SHA256 hash of the request body (or empty string for GET).

Session Cookie (Web UI)

For web applications, login via /web/login to obtain a session cookie:

curl with session cookie
$ curl -b "session_token=abc123..." \
http://pulldb-server:8080/api/jobs
Note: Most users should use the pulldb CLI rather than calling the API directly. The CLI handles HMAC signing automatically using PULLDB_API_KEY and PULLDB_API_SECRET environment variables.
Security: The X-Trusted-User header is no longer supported and will be rejected. All requests must use HMAC signatures or session cookies.

Register User

Self-register a new user account. No authentication required.

POST /api/auth/register

Request Body

{
  "username": "jsmith",
  "password": "securepassword123",
  "host_name": "dev-workstation"
}
Field Type Required Description
username string Yes Username (3-50 chars, alphanumeric + underscore)
password string Yes Password (minimum 8 characters)
host_name string No Hostname for API key (auto-detected by CLI)

Response (201 Created)

{
  "username": "jsmith",
  "user_code": "JSMITH",
  "api_key": "key_abc123def456...",
  "api_secret": "sec_xyz789...",
  "message": "Account created successfully. Contact an administrator to enable your account."
}
Pending Approval: New accounts are created in a disabled state. An administrator must enable your account AND approve your API key before you can submit jobs.

Request Host Key

Request an API key for a new host machine. Requires password authentication.

POST /api/auth/request-host-key

Request Body

{
  "username": "jsmith",
  "password": "securepassword123",
  "host_name": "my-laptop"
}
Field Type Required Description
username string Yes Your existing username
password string Yes Your current password
host_name string Yes Hostname of the new machine (max 255 chars)

Response (200 OK)

{
  "username": "jsmith",
  "user_code": "JSMITH",
  "host_name": "my-laptop",
  "api_key": "key_def456ghi789...",
  "api_secret": "sec_abc123...",
  "message": "API key created successfully. Contact an administrator to approve the key before it can be used."
}
Use Case: Use this endpoint when you already have a pullDB account but need to access pullDB from a different machine. The CLI command pulldb register calls this endpoint automatically when it detects you're an existing user on a new machine.

Submit Job

Create a new restore job. Requires HMAC authentication.

POST /api/jobs

Request Body

{
  "user": "jsmith",
  "customer": "acme_pest",
  "dbhost": "localhost",
  "date": "2026-01-01",
  "overwrite": false
}
Field Type Required Description
user string Yes User code for job ownership (e.g., "jsmith")
customer string Yes* Customer name matching S3 backup path
qatemplate boolean Yes* Use QA template backup instead (default: false)
dbhost string No Target database host (default from user settings)
date string No Backup date (YYYY-MM-DD), defaults to latest
overwrite boolean No Overwrite existing database (default: false)
suffix string No 1-3 lowercase letter suffix for target (e.g., "dev")
custom_target string No Custom target database name (1-51 lowercase letters, pattern: ^[a-z]{1,51}$)
backup_path string No Full S3 path to specific backup (overrides customer/date lookup)
env string No S3 environment: "staging" or "prod" (default: searches both)

* Either customer or qatemplate required, not both

Custom Target + Suffix: custom_target and suffix cannot be used together. Include any suffix directly in your custom target name.

Response (201 Created)

{
  "job_id": "7f3a2b1c-85a1-4da2-9c8e-1234567890ab",
  "target": "jsmithacmepest",
  "staging_name": "pulldb_stg_7f3a2b1c",
  "status": "queued",
  "owner_username": "jsmith",
  "owner_user_code": "jsmith",
  "submitted_at": "2026-01-02T10:42:15Z"
}

Example

curl
$ curl -X POST \
-H "X-API-Key: $PULLDB_API_KEY" \
-H "X-Timestamp: 2026-01-03T15:42:00Z" \
-H "X-Signature: $SIGNATURE" \
-H "Content-Type: application/json" \
-d '{"user":"jsmith","customer":"acme_pest"}' \
http://pulldb-server:8080/api/jobs

Cancel Job

Cancel a queued or running job. Requires authentication.

POST /api/jobs/{job_id}/cancel

Response (Queued Job)

{
  "job_id": "7f3a2b1c-85a1-4da2-9c8e-1234567890ab",
  "status": "canceled",
  "message": "Job canceled successfully (was queued)"
}

Response (Running Job)

{
  "job_id": "7f3a2b1c-85a1-4da2-9c8e-1234567890ab",
  "status": "canceling",
  "message": "Cancellation requested; worker will stop at next checkpoint"
}
Returns 409 Conflict if job is already in terminal state (complete, failed, canceled), or if the job has entered the loading phase (myloader cannot be safely interrupted). Users can only cancel their own jobs unless they have manager or admin privileges. Returns 401 Unauthorized if authentication headers are missing.

Extend Retention

Extend the retention period for a job's database. Requires authentication.

POST /api/jobs/{job_id}/extend

Request Body

{
  "months": 3
}
Field Type Required Description
months integer No Months to extend (default: 1, max: 12)

Response

{
  "success": true,
  "message": "Retention extended by 3 month(s)",
  "job_id": "7f3a2b1c-85a1-4da2-9c8e-1234567890ab",
  "expires_at": "2026-04-02T10:42:15Z",
  "locked_at": null,
  "locked_by": null
}
Authorization: Users can only extend their own jobs. Administrators can extend any job.

Lock Database

Lock a job's database to prevent automatic cleanup. Requires authentication.

POST /api/jobs/{job_id}/lock

Request Body

{
  "reason": "Keeping for demo next week"
}
Field Type Required Description
reason string No Reason for locking (default: "Locked via API")

Response

{
  "success": true,
  "message": "Database locked successfully",
  "job_id": "7f3a2b1c-85a1-4da2-9c8e-1234567890ab",
  "expires_at": "2026-04-02T10:42:15Z",
  "locked_at": "2026-01-05T14:30:00Z",
  "locked_by": "jsmith"
}
Important: Locked databases are protected from automatic retention cleanup and cannot be overwritten. Remember to unlock when no longer needed.

Search for available customer backups in S3. Authentication optional but recommended.

GET /api/backups/search

Query Parameters

date_from string No Filter from date (YYYYMMDD). Default: 7 days ago
limit integer No Max results per page (default: 5, max: 100)

HTTP Status Codes

Code Meaning
200 Success
201 Created (job submitted)
400 Bad Request (invalid parameters)
401 Unauthorized (missing/invalid authentication)
403 Forbidden (insufficient permissions)
404 Not Found (job/backup doesn't exist)
409 Conflict (e.g., can't cancel completed job)
500 Internal Server Error

Error Handling

Error responses use standard FastAPI/HTTP format with a detail field:

{"detail": "No backup found for customer 'acme' on date '2026-01-01'"}

For validation errors (422), the response includes field-level details:

{
  "detail": [
    {
      "type": "string_type",
      "loc": ["body", "target"],
      "msg": "Input should be a valid string",
      "input": null
    }
  ]
}

Common Error Messages
