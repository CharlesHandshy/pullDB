Jobs Management

The Jobs page lets you view, monitor, and manage your restore jobs. Track progress in real-time and view detailed logs for each job.

Overview

The Jobs page provides a comprehensive view of all your restore jobs. You can see active jobs updating in real-time, browse job history, and drill down into details.

Job List

Sortable, filterable table of all jobs.

Search & Filter

Find jobs by target database, status, or date.

Real-Time Updates

Live progress for running jobs.

Detailed Logs

Full execution logs for each job.

Jobs List

The main jobs table displays all your jobs with key information at a glance.

Two views
The Jobs page has two tabs: Active (queued, running, canceling, and deployed jobs) and History (failed, canceled, dropped, and superseded jobs). The Active view is shown by default and refreshes automatically.

Table Columns

Column Description Sortable
Job ID Unique identifier for the job (first 8 characters shown) Yes
Target Target database name the restore is writing to Yes
Status Current job state with badge Yes
Host Database host the restore is running on Yes
Submitted Date the job was submitted Yes
Actions View details and cancel buttons (role-dependent) No

Virtual Scroll

The jobs table uses virtual (infinite) scroll — 50 jobs are loaded initially, and more are fetched automatically as you scroll to the bottom. There are no page controls.

Filtering & Search

Use the filter bar above the jobs table to narrow down results.

Available Filters

Filter Description
Search Free text search across job ID and target database name
Status Filter by job state: All, Queued, Running, Complete, Failed, Canceled
Date Range Filter jobs submitted within a date range
Target Database Filter by target database name (filter_target)
Quick Filters
Click on any status badge in the table to quickly filter by that status.

Keyboard Shortcuts

/ Focus search field
R Refresh job list
Esc Clear filters

Job Details

Click on any job row (or the "View" button) to open the job details page. This provides comprehensive information about the job.

Queued and running jobs
Queued jobs show their position in the queue. Running jobs display a phase stepper and live execution log that auto-refreshes every 2 seconds.

Details Page Sections

Header

Job ID, target database, status badge, and quick action buttons.

Progress

Visual progress bar showing phases: Download → Restore → Post-SQL.

Details

Full job configuration: backup date, target database, host, options.

Logs

Expandable log viewer with real-time streaming for active jobs.

Progress Phases

For running jobs, the progress bar shows which phase is currently executing:

  • DISCOVERY — Finding the best backup in S3
  • DOWNLOAD — Downloading backup from S3
  • EXTRACTION — Decompressing backup archive
  • MYLOADER — Restoring database with myloader
  • POST_SQL — Running post-restore SQL scripts
  • ATOMIC_RENAME — Switching staging to target database

Connection Info

For jobs in the deployed or complete state, the details page shows connection information for the restored database:

Connection Details
Host: staging-mysql.example.com
Port: 3306
Database: ab1234acmecorp
User: (your credentials)

Job Actions

Depending on job status, different actions are available.

Available job actions and when they can be used
Action Available When Description
View Always Open job details page
Cancel Queued, Running Cancel the job (with confirmation)
Resubmit Failed (History view) Create a new job using the same backup and settings
Delete Deployed, Failed (History view) Delete job databases and/or records
Copy ID Always Copy job ID to clipboard
Cancel During Restore
If you cancel a job during the RESTORE phase, the partially restored database will be automatically dropped to avoid leaving inconsistent data.

Resubmitting Failed Jobs

When a job fails, you can resubmit it using the same backup and settings. The resubmit button appears in the History view for failed jobs you have permission to act on.

Permission Model
  • Users can resubmit their own failed jobs
  • Managers can also resubmit jobs from users they manage
  • Admins can resubmit any failed job
When a manager or admin resubmits, the new job is created as the original owner.

Resubmit may be blocked or show warnings in certain situations:

Situations that block or warn when resubmitting a job
Situation Result
Active job exists for same target Blocked — wait for it to complete or cancel it
Deployed job exists (different owner) Blocked — only admins can override
Deployed job exists (same owner) Warning — will overwrite existing database
Host is disabled Warning — job will queue but won't run

Status Badges

Jobs display color-coded status badges for quick identification. There are 11 possible states.

Active States

Jobs in these states are still being processed:

queued ⏳

Job is waiting in the queue for a worker to pick it up.

running ⚙️

Job is actively being processed. Check the phase for current step.

canceling 🛑

Cancel requested. Worker is cleaning up resources before terminating.

deleting 🗑️

Database deletion in progress. The staging database is being dropped.

Terminal States (Success)

Jobs in these states have reached a successful terminal state:

complete ✅

Job finished successfully. Database is ready to use.

deployed 🚀

Database is locked (protected from retention). Use this for long-term test environments.

Terminal States (Cleanup)

Jobs in these states are no longer active:

expired ⏱️

Database exceeded retention period and was automatically deleted.

superseded 🔄

A newer restore for the same target replaced this job's database.

deleted 🗑️

Database was manually deleted by user request.

Terminal States (Failure)

Jobs that did not finish successfully:

failed ❌

Job encountered an error. Check logs for details.

canceled 🚫

Job was canceled by user before finishing.

Database Lifecycle

Restored databases have a lifecycle managed by the retention system. Understanding this helps you manage your staging databases effectively.

Retention System

By default, staging databases are automatically deleted after the retention period (typically 7 days). This ensures resources are cleaned up and prevents stale data.

Event Effect
Retention period expires Job transitions to expired, database is deleted
New restore to same target Previous job becomes superseded, database is replaced
User deletes database Job transitions to deleted

Locking Databases

To protect a database from automatic deletion, you can lock it. Locked databases remain in deployed status and are excluded from retention.

Lock vs Complete
complete jobs will be cleaned up by retention. deployed jobs are protected until manually unlocked or deleted.

Supersession

When you restore a customer to the same target database that already exists:

  • The existing database is dropped (if Overwrite is enabled)
  • The previous job is marked as superseded
  • Only the new job shows as the active restore for that target
Locked Databases Cannot Be Superseded
If the existing database is locked (deployed), the new restore will fail unless you unlock or delete the existing database first.
