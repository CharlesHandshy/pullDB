Admin Panel

The Admin Panel provides system-wide configuration and management capabilities. Only users with the Admin role can access these features.

Admin Access Required
This page documents features only available to Admin users. If you don't see the Admin menu item, contact your system administrator to request access.

Overview

The Admin Panel is your central hub for managing the pullDB system. It is organized into separate sub-pages, each accessible from the Administration menu:

Admin Sub-Pages
/web/admin/users — User management
/web/admin/hosts — Host configuration
/web/admin/keys/pending — API key approvals
/web/admin/disallow — Disallowed usernames
/web/admin/admin-tasks/{task_id} — Background task status

Users

Enable, disable, and manage user accounts and roles.

Hosts

Configure MySQL hosts for restores.

API Keys

Approve pending keys and revoke access system-wide.

Settings

System-wide configuration options.

User Management

The Users section (/web/admin/users) lets you manage all user accounts in the system.

User List

The user list is a sortable, filterable table displaying all accounts. You can sort by username, user code, role, status, creation date, and last login. Filter by role (user, manager, admin) or by status (enabled, disabled). Click any row to open the user detail page.

Column Description
Username Unique login identifier
User Code Short code used in database name prefixes
Role user, manager, or admin
Status Enabled or Disabled
Last Login Most recent authentication timestamp

User Roles

Role Description
user Standard user — can only manage their own jobs
manager Can view all users' jobs and create new users
admin Full administrative access

User Actions

From the user detail page you can:

  • Enable / Disable — Toggle user access
  • Force Password Reset — Require a password change on next login
  • Set Role — Assign user, manager, or admin role
  • Set Manager — Assign a manager for the user
  • View Jobs — See this user's job history and active jobs
Reset Password
Use the "Reset Password" action to generate a new temporary password for a user. The password will be displayed once — make sure to communicate it securely.

New User Registration Flow

New users register via the CLI (pulldb register) or the web registration form. Accounts are created in a disabled state and cannot use the CLI until two separate steps are completed:

  1. Enable the user — Administration → Users → Enable
  2. Approve their API key — Administration → API Keys → Approve

This two-step process is intentional: user activation and host authorization are separate security decisions.

Force Delete User

Force Delete is an async background operation available from the user detail page. It is used to permanently remove a user and all their data from the system. It is distinct from simply disabling an account.

What Force Delete Does
Force Delete runs as an admin task in the background worker. It:
  • Drops each of the user's staged databases on the target hosts
  • Deletes all of the user's job records
  • Deletes the user account itself
  • Logs all actions to the audit log
You can optionally skip database drops if you only want to remove the account and job records. After submission, the UI redirects to the task status page (/web/admin/admin-tasks/{task_id}) which auto-polls until completion.
Force Delete is Irreversible
Staged databases are permanently dropped. Only use Force Delete when you are certain the user's data is no longer needed. To preserve job history while removing access, use Disable instead.

Host Management

The Hosts section (/web/admin/hosts) configures which MySQL servers are available as restore targets.

Host List

The host list shows each registered server with:

Column Description
Hostname / Alias Server address and friendly alias
Status Enabled or disabled for new restores
Max Concurrent Jobs How many restores can run at once on this host
Active Jobs Jobs currently running on this host
Last Successful Restore Timestamp of the most recent completed restore
Credential Status Whether the AWS Secrets Manager reference is valid

Adding a Host

There are two ways to add a host:

  • Add Host (simple) — For hosts already provisioned with a pulldb_loader MySQL user and credentials stored in AWS Secrets Manager. Provide the hostname/alias, the Secrets Manager reference, and the max concurrent jobs limit.
  • Provision New Host (/web/admin/hosts/provision) — Full wizard that creates the MySQL user, deploys the required stored procedure, stores credentials in Secrets Manager, and registers the host in one guided flow.
Credential Storage
pullDB does not store MySQL passwords directly. Credentials are held in AWS Secrets Manager and referenced by a path (e.g., aws-secretsmanager:/pulldb/mysql/dev-db-01). The host detail page shows the reference path, not the password.

Host Detail Page

Click a host to open its detail page (/web/admin/hosts/{host_id}), which shows:

  • Connection information and credential reference
  • Active jobs on this host
  • Historical job statistics
  • Stored procedure version
  • A Rotate Credentials button to atomically update the MySQL password and the Secrets Manager secret

API Key Management

The API Keys section (/web/admin/keys/pending) shows all pending keys awaiting approval. Every key is tied to a specific host and requires admin approval before the CLI can use it.

Key States

State Description CLI Access
pending Awaiting admin approval Blocked
approved Admin approved, active Works
revoked Admin revoked Blocked

Key Actions

Action Description
Approve Authorize the key — CLI access is granted immediately
Revoke Immediately invalidate an approved key; no grace period
Audit Trail
Key approvals and revocations are logged to the audit log with the admin's username and timestamp. IP addresses for key creation and last use are also tracked.
Two-Step Activation for New Users
For brand-new users there are two separate steps: (1) enable the user account, and (2) approve their API key. Both must be done before the CLI will work. Enabling the user without approving the key leaves CLI access blocked.

System Settings

The Settings section contains global configuration options. Settings can be stored in the database, in the .env file, or both. The priority order is: Environment > Database > Default.

Key Settings

System configuration settings with defaults and descriptions
Setting Default Description
max_active_jobs_per_user 0 (unlimited) Per-user concurrent job limit
max_active_jobs_global 0 (unlimited) System-wide concurrent job limit
default_retention_days 7 Default expiration for new restores
max_retention_days 180 Maximum retention a user can request
staging_retention_days 7 Days before staging databases are eligible for cleanup
cleanup_grace_days 7 Days after expiry before automatic cleanup runs
expiring_warning_days 7 Days before expiry to show the expiration warning
job_log_retention_days 30 Days before job event logs are pruned
myloader_threads 4 Parallel restore threads
myloader_timeout_seconds 86400 Maximum myloader execution time (24 hours)
maintenance_mode false Global maintenance mode (database-only setting)
Settings Take Effect Immediately
Database settings are picked up on the next worker poll cycle — no restart is required. Environment variable overrides in .env require a service restart to reload. Use pulldb-admin settings diff to check for drift between database and .env values.

Maintenance Mode

Maintenance mode stops the worker from claiming new jobs while allowing any currently running jobs to finish gracefully. It is the safest way to pause the system for upgrades or emergency interventions.

How to Toggle Maintenance Mode
Go to Administration → Settings and toggle the maintenance_mode setting, or use the CLI:
  • pulldb-admin maintenance enable
  • pulldb-admin maintenance disable
  • pulldb-admin maintenance status
Maintenance mode is stored as a database-only setting and takes effect on the worker's next poll cycle — no restart is needed.

Cleanup Tools

Several cleanup operations are available from the Administration menu:

Prune Logs

Remove old job event log entries to control database size. Available at /web/admin/prune-logs/preview. Supports a dry-run preview before committing.

Cleanup Staging

Drop orphaned staging databases from failed or interrupted jobs. Available at /web/admin/cleanup-staging/preview. Supports a dry-run preview and an age filter.

Orphan Databases

Databases matching the staging pattern that have no corresponding job record. Available at /web/admin/orphans. Supports per-host filtering, single delete, and bulk delete.

User Orphan Databases

Databases restored by users who have since been removed from the system. Available at /web/admin/user-orphans. Manual cleanup is required.

Audit Log

All significant administrative actions are recorded in the audit log (/web/admin/audit/). Logged events include user enable/disable, role changes, API key approvals and revocations, host additions and removals, setting changes, and orphan database deletions. The log is filterable by action type, admin username, and date range.

Background Tasks (Admin Tasks)

Admin Tasks are asynchronous background operations that handle long-running or bulk administrative actions without blocking the UI. They are executed by the worker process and can be monitored from the task status page (/web/admin/admin-tasks/{task_id}).

Task Types

Task Description Launched From
force_delete_user Drop user's staged databases, delete job records, and remove the account Administration → Users → (user detail) → Force Delete
scan_user_orphans Find databases on hosts that belong to a deleted user Administration → User Orphans → Scan
bulk_delete_jobs Delete multiple jobs and their databases in one operation Job selection interfaces
retention_cleanup Drop all expired databases that are past the grace period Scheduled by the worker

Task States

pending running complete failed

Tasks move from pending to running to either complete or failed. There is no cancellation mechanism — tasks run to completion or failure.

Task Status Page

After a task is created, the UI redirects to its status page at /web/admin/admin-tasks/{task_id}. The page auto-polls while the task is pending or running, then stops once a terminal status is reached. It shows:

  • Task type and current status badge
  • Parameters (target user, databases to drop, etc.)
  • Result JSON on completion
  • Error detail on failure
Single-Task-Per-Type Enforcement
The system prevents two concurrent tasks of the same type. For example, you cannot run two force_delete_user tasks simultaneously. If you try, the UI returns an error asking you to wait for the in-progress task to finish. A force_delete_user and a retention_cleanup can coexist, but the worker processes tasks one at a time.
Orphan Recovery
If a task is stuck in running state (e.g., because the worker crashed), the worker will automatically reclaim it after a 10-minute timeout. A warning is logged when reclaiming occurs.

Disallowed Users

The Disallowed Users page (/web/admin/disallow) manages a block list of usernames that cannot be registered. This prevents users from creating accounts with system-reserved names or usernames you want to exclude for policy reasons.

Two Types of Entries

Type Description Removable
Hardcoded System-reserved names built into the application (e.g., admin, root, test) No
Custom Admin-added usernames stored in the database Yes

Adding a Disallowed Username

In the Web UI, click Add Disallowed User, enter the username, and optionally provide a reason. From the CLI:

# Add (reason defaults to "Admin-configured" if omitted)
pulldb-admin disallow add baduser --reason="Policy violation"

# List all entries (hardcoded + custom)
pulldb-admin disallow list

# Remove a custom entry
pulldb-admin disallow remove baduser

# Skip confirmation prompt
pulldb-admin disallow remove baduser --force
Applies at Registration Only
The disallow list is checked when a new account is created. It does not affect existing accounts. If a user already has an account with a username you later add to the list, their account is not automatically disabled.

Locked Databases

The Locked Databases page (/web/admin/locked-databases) shows all deployed databases that users have locked to prevent automatic expiration. Admins can view lock details and override locks if necessary.

Lock Information

Column Description
Target The database name that is locked
Host MySQL server where the database resides
Locked By Username who locked the database
Locked At When the lock was applied
Reason User-provided reason for locking (if any)

Admin Unlock

Admins can forcibly unlock a database if needed. This removes the lock and allows normal retention and cleanup to proceed.

Communicate Before Unlocking
Before forcibly unlocking a database, contact the lock owner to understand why they locked it. Unlocking may cause unexpected data loss if the database expires.
