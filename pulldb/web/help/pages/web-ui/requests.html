Feature Requests

The Feature Requests page allows all users to suggest improvements, vote on existing ideas, and track the status of requested features. Access it from the sidebar under "Resources".

All Users Welcome
Feature Requests is available to all authenticated users. You can browse, submit, and vote on requests regardless of your role (user, manager, or admin).

Overview

The Feature Requests system helps the pullDB team understand what improvements matter most to users. By voting on requests, you help prioritize development efforts.

Page Layout
The Feature Requests page shows four statistics cards at the top (Open, In Progress, Complete, Declined counts), a + New Request button, and a sortable/filterable request table below. Click any row to expand the full detail panel with voting controls, notes, and — if you are the primary admin — status management fields.

Page Components

  • Statistics Cards — Overview of request counts by status (Open, In Progress, Complete, Declined)
  • Submit Button — Opens the new request form
  • Request Table — Sortable, filterable list of all requests
  • Vote Controls — Upvote requests you'd like to see implemented

Browsing Requests

The request list displays all submitted feature requests with key information. Use the table controls to find requests you're interested in.

Table Columns

Column Description
Votes Total vote count with upvote button — click to vote
Title Request title — click to expand and view details
Status Current status (Open, In Progress, Complete, Declined)
Submitted By Username and user code of the person who submitted
Created When the request was submitted

Sorting & Filtering

  • Sort by Votes — Default sort shows most popular requests first
  • Sort by Date — Click "Created" header to see newest requests
  • Filter by Status — Use the status column filter to show only Open, Complete, etc.

Submitting a Request

Have an idea for improving pullDB? Submit a feature request to share it with the team and other users.

How to Submit

  1. Click the "Submit Request" button in the top-right corner
  2. Enter a clear, descriptive title (5-200 characters)
  3. Optionally add a description with more details (up to 2000 characters)
  4. Click "Submit" to create the request
Before Submitting
Search existing requests first to avoid duplicates. If a similar request exists, vote for it instead of creating a new one.

Writing Good Requests

Element Good Example Poor Example
Title "Add email notifications for job completion" "Notifications"
Description "Send email when restore completes, with database name and connection details" "Would be nice to have"

Voting

Voting helps prioritize which features get implemented first. Each user can cast one vote per request — upvote, downvote, or no vote. The displayed score is upvotes minus downvotes.

How Voting Works

  • One vote per request — You can upvote or downvote each request independently
  • Upvote — Send vote_value: 1 via the upvote button to support a request
  • Downvote — Send vote_value: -1 via the downvote button if you disagree
  • Remove vote — Send vote_value: 0 (click your active vote again) to retract it
  • Vote indicator — The UI highlights your current vote on each request

Vote API

The voting API requires an explicit vote_value integer. There is no toggle — you must send the exact value you want:

vote_valueEffect
1Upvote (adds 1 to score)
-1Downvote (subtracts 1 from score)
0Remove your existing vote

The response includes the updated vote_score, upvote_count, downvote_count, and your current user_vote (an integer: 1, -1, or null when not voted).

Request Status

Each request progresses through a workflow. Only the primary administrator (the first installed admin account) can change status or set the official admin response — both via the web UI and via the API endpoint (POST /web/requests/api/update/{request_id}).

open

New request awaiting review. Can receive votes.

in progress

Request approved and actively being worked on.

complete

Feature has been implemented and released.

declined

Request was not approved. Admin response explains why.

Admin Response

When the primary administrator updates a request's status, they may set an official admin response — a single text field that appears prominently in the request detail panel. This is especially useful for declined requests, where the response explains why the feature will not be implemented, or for in-progress requests to reference a target release version.

The admin_response field is separate from the notes thread. Notes can be added by any authenticated user; the admin response is set only by the primary administrator.

Adding Notes

Click on any request to expand its details and view or add notes. Notes allow any authenticated user to discuss or provide context on a request. Notes are separate from the official admin response field.

Note Features

  • View Notes — Click a request row to expand and see all notes
  • Add Note — Any authenticated user can type in the note field and submit
  • Delete Note — You can delete your own notes; admins can delete any note
  • Timestamps — Notes show when they were posted and by whom
Cannot Edit Requests
There is no edit endpoint for feature requests. If you need to correct a submitted request, add a clarifying note, or delete the original and resubmit with the corrected content.

Withdrawing Your Request

If you submitted a request and no longer want it tracked, you can withdraw it.

How to Withdraw

  1. Find your request in the list
  2. Click to expand the request details
  3. Click the "Withdraw" button
  4. Confirm the withdrawal
Permanent Action
Withdrawing a request permanently deletes it along with all notes and votes. This cannot be undone.

Exporting Requests (Admin Only)

Administrators can download a full JSON bundle of all feature requests, including votes and notes, using the export endpoint: 

GET /web/requests/api/export?status=open,in_progress

The status query parameter is optional; omit it to export all requests regardless of status. The response is a downloadable JSON file containing requests with their complete vote counts and note threads. Admin role is required.
