Restore Workflow

The restore page is a single-page form at /web/restore. It has two numbered sections: Source Selection and Target Configuration. Fill them in order and click Queue Restore to submit the job.

~1 minute to submit

Overview

Click New Restore in the sidebar to open the form. The page is divided into two sections that you complete from top to bottom. There is no multi-step wizard and no separate confirmation screen — everything is on one page.

Section 1: Source Selection

The Source Selection section has two tabs: Customer Database (the default) and QA Template. The Customer Database tab is described below; the QA Template tab is covered in its own subsection.

Customer Search

Before searching for a customer, you can optionally filter which backup dates will be shown. The date filter defaults to the last 7 days. Use the mode selector to switch between On or After, On or Before, On Date, and Between. Click the reset button (↺) to return to the 7-day default.

Type at least 3 characters into the Customer search box. Results appear as you type — the search is case-insensitive and backed by a live JSON endpoint at /web/restore/search-customers. Wildcard characters (* and ?) are supported for flexible matching. Click a result row to select that customer.

Search is case-insensitive
Customer identifiers are normalized to lowercase internally. You can type in any case — the search will match regardless. For example, typing ACME will find acme_pest and acmeplumbing.

Once a customer is selected, the field collapses to show the selected name with a Change link. If the customer name is very long (over 42 characters), a normalization notice appears explaining the truncated name that will be used in the target database name.

Backup Picker

After a customer is selected, the Backup list loads automatically via HTMX from /web/restore/search-backups. The list shows available backups within your chosen date range, with the most recent listed first. Each entry shows:

Column Description
Date Backup date (YYYY-MM-DD)
Size Compressed backup size (e.g., 1.2 GB)
Environment Source environment (prod, staging, etc.)

Click a row to select a backup. The row highlights and the backup field collapses to a summary with a Change link. Selecting a backup also causes the Target Configuration section to display the auto-generated target database name preview.

No backups shown?
Widen the date filter or click the reset button to return to the last 7 days. If the list is still empty, the customer may have no backups in S3 for that period. Check with your administrator that the S3 bucket is correctly configured.

QA Template Tab

The QA Template tab is used to restore the shared QA template database — a pre-sanitized dataset for testing and development purposes. Switching to this tab skips the customer search entirely and shows only the backup list for the QA template dataset.

The date filter works identically to the Customer tab. Select a QA template backup from the list to proceed.

QA Template target name
When using the QA Template tab, the target database name is auto-generated as {user_code}qatemplate. For example, if your user code is ab1234, the target will be ab1234qatemplate. The Suffix field is hidden; the Overwrite toggle remains available.

Section 2: Target Configuration

The Target Configuration section controls where the restore lands and what the target database will be named. It becomes fully visible once a backup is selected in Section 1.

Target configuration fields for the restore form
Field Required Description Default
Database Host Yes Target MySQL server. Dropdown shows only hosts assigned to your account. Your default host (if set in profile)
Suffix No 1–3 lowercase letters appended to the auto-generated database name (e.g., dev). None
Custom toggle No Override the auto-generated name with a custom name (lowercase letters only, 1–51 characters). Mutually exclusive with Suffix. Off (auto-generated)
Overwrite toggle No If a database with the target name already exists, delete it and restore fresh. A warning appears when enabled. Off

Target Database Naming

The target name preview updates live as you make selections. The auto-generated name follows this pattern:

Auto-generated target database name formats by scenario
Scenario Format Example
Customer, no suffix {user_code}_{customer} ab1234_acmepest
Customer, with suffix {user_code}_{customer}{suffix} ab1234_acmepestdev
QA Template {user_code}qatemplate ab1234qatemplate
Custom name your input myprojectdb
Long customer names are normalized
Customer names longer than 42 characters are truncated to 38 characters and a 4-letter deterministic hash suffix is appended. The normalization notice in Section 1 shows you the result before you submit. The same long customer name always produces the same normalized form.
Overwrite permanently deletes the existing database
When the Overwrite toggle is enabled, any database currently at the target name will be dropped before the restore runs. This cannot be undone. Do not enable Overwrite unless you are certain you no longer need the existing data.

Host Selector

The Database Host dropdown shows only the MySQL servers your account is authorized to use. If your administrator has set a default host on your profile, it will be pre-selected. If no hosts are shown, contact your manager or administrator to request access.

Submitting on behalf of another user (Managers and Admins)
Navigate to /web/restore?for_user=true to see an additional Submit as User section. Choose a managed user from the dropdown; the job will appear in their job list and use their user code for target naming.

After Submission

Click Queue Restore to submit the form. On success, you are redirected to the Jobs page (/web/jobs) where your new job appears at the top. From there you can click the job row to open the detail page and watch progress in real time.

Override acknowledgment
If the target name matches a known customer name, or if the target database is currently owned by another user, the form will not submit immediately. Instead, it shows a Please Confirm panel with checkboxes you must tick before re-submitting. Read each acknowledgment carefully — confirming an ownership transfer supersedes the other user's job and creates an audit record.

Job Phases

Once queued, a restore moves through these phases. The job detail page at /web/jobs/{job_id} shows a phase stepper that updates every 2 seconds while the job is active:

1. queued

Job is waiting for a worker. Usually only a few seconds.

2–7. running

Worker progresses through: Discovery → Download → Extraction → Loading → Post-SQL → Rename.

8. deployed

Restore complete. The database is live; connection details appear on the job page.

  • Phase stepper — eight phases shown in order; completed phases show a checkmark, the current phase shows a spinner
  • Auto-refresh — the status header and phase stepper poll every 2 seconds; no manual page reload needed
  • Execution log — timestamped events with virtual scrolling for large logs; new entries appear at the top while running
  • Pause/resume — a toggle button stops or restarts auto-polling if you want to read the log without it jumping
  • Completion — when the job reaches the complete event, the page reloads automatically to show connection details (hostname, database name, port)

Restore duration depends on backup size and host load. A 1 GB backup typically completes in 3–10 minutes. Larger backups can take longer — the log shows per-phase timing once the job finishes.

Common Mistakes

Leaving the target as auto-generated when you need a specific name

The auto-generated name is predictable: {user_code}_{customer}. If two restores of the same customer are submitted without a suffix or custom name, the second job will supersede the first and you will lose the first database. Use the Suffix field (e.g., v2) or enable Custom to give each restore a distinct name when you need both.

Enabling Overwrite on a database you are still using

The Overwrite toggle drops the existing database before the restore starts. If the database is connected to an active service or another user is relying on it, it will be gone with no recovery path from pullDB. Disable Overwrite (the default) unless you are certain the existing database can be deleted.

No database hosts available

If the Database Host dropdown is empty, your account has not been assigned any target hosts. Contact your manager or an administrator to request access. You cannot submit a restore until at least one host is assigned to your account.

Backup list is empty after selecting a customer

The default date filter covers the last 7 days. Click the reset button (↺) to confirm you are using the default range, then try widening it with a broader On or After date. If backups still do not appear, the customer may have no backups in S3, or there may be a connectivity issue with the backup bucket. Check with your administrator.
