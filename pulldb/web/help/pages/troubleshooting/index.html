Download Issues

Symptoms: Job fails in download phase with S3 access error.

Causes:

  • AWS credentials expired or missing
  • IAM role doesn't have S3 read permissions
  • Wrong S3 bucket or path configured

Solutions:

  1. Check your AWS credentials are current:
    $ aws sts get-caller-identity
  2. Verify S3 access:
    $ aws s3 ls s3://your-bucket/backups/
  3. Contact your admin to verify IAM permissions

Symptoms: Job fails immediately with "backup not found" message.

Causes:

  • Customer ID misspelled
  • Backup doesn't exist for requested date
  • Backup was deleted or moved

Solutions:

  1. Verify available backups:
    $ pulldb list <customer>
  2. Double-check customer ID spelling
  3. Try without date= to get latest backup

Symptoms: Download starts but times out before completing.

Causes:

  • Network connectivity issues
  • Very large backup file
  • S3 throttling

Solutions:

  1. Check network connectivity
  2. For large backups, expect longer times (check pulldb events for progress)
  3. Retry the job - transient network issues often resolve

Restore Issues

Symptoms: Job fails in myloader phase with CREATE DATABASE error.

Causes:

  • MySQL staging user lacks CREATE privilege
  • Database name collision

Solutions:

  1. Contact admin to verify staging user permissions
  2. Check for orphaned staging databases:
    $ pulldb-admin cleanup --dry-run

Symptoms: Job fails in atomic_rename phase because target database exists.

Solution: Add overwrite to replace the existing database:

$ pulldb restore acme overwrite
Warning: overwrite will delete the existing database. Make sure you don't need it!

Symptoms: Job fails in post_sql phase.

Causes:

  • Schema changed and script references missing columns
  • Script has SQL syntax errors
  • Large UPDATE times out

Solutions:

  1. Check job events for the exact error:
    $ pulldb events <job_id>
  2. Contact admin to review/update post-SQL scripts

Permission Issues

Symptoms: Can't submit jobs or run commands.

Causes:

  • Username not registered in system
  • User account disabled

Solutions:

  1. Verify your username is correct
  2. Contact admin to enable your account

Symptoms: Job completes but you can't access the database.

Causes:

  • MySQL user doesn't have GRANT on the restored database
  • Host restriction on MySQL user

Solutions:

  1. Verify database exists:
    $ mysql -e "SHOW DATABASES LIKE '%yourcode%'"
  2. Contact admin to verify MySQL permissions

Connection Issues

Symptoms: CLI commands fail with connection errors.

Causes:

  • pullDB service not running
  • Firewall blocking port
  • Wrong API URL configured

Solutions:

  1. Check your configuration:
    $ cat ~/.pulldb/config
  2. Verify service is reachable
  3. Contact admin if service is down

Job Stuck

Symptoms: Job stays in queued for more than a few minutes.

Causes:

  • No workers running
  • All workers busy with other jobs
  • Worker crashed

Solutions:

  1. Check how many jobs are ahead of you:
    $ pulldb status
  2. If stuck for >5 minutes with no queue, contact admin

Symptoms: Job shows running but no progress for a long time.

Solutions:

  1. Check events for the current phase:
    $ pulldb events <job_id> --follow
  2. download - Large backup, check progress %
  3. myloader - Large database, can take 10+ minutes
  4. If truly stuck, cancel and retry:
    $ pulldb cancel <job_id>
Cancellation is not instantaneous. When a job is in running state, canceling it transitions the job to canceling first. The worker will finish the current checkpoint operation (for example, a myloader import in progress), then stop cleanly. This can take up to 30 seconds. Do not force-kill the container while canceling.

Slow Restore

Expected restore times
Small databases (<1GB): 2-5 minutes
Medium databases (1-5GB): 5-15 minutes
Large databases (5-20GB): 15-45 minutes

If your restore is slower than expected:

  • Check which phase is taking longest with pulldb events
  • Downloading slow: Network or S3 throughput issue
  • Restoring slow: MySQL I/O or table complexity (indexes)
  • Contact admin if consistently slower than expected

Disk Space Issues

Symptoms: Job fails with disk space errors.

Space requirements:

  • Download: Size of compressed backup
  • Extract: 2-3x compressed size
  • Total: ~4x compressed backup size

Solutions:

  1. Check available space on the pullDB work volume:
    $ df -h /mnt/data
  2. Backup archives and extracted files accumulate in /mnt/data/pulldb/work/. Completed jobs clean up automatically, but failed or orphaned jobs may leave artifacts behind.
  3. Ask your admin to run staging cleanup to reclaim orphaned directories:
    $ pulldb-admin cleanup --dry-run
  4. If the volume is consistently full, contact your administrator to expand the /mnt/data storage allocation.

Authentication Errors

Symptom: You are redirected to the login page with a "Session expired" message.

Cause: Session tokens expire after 24 hours by default.

Fix: Log in again. If your team needs longer sessions, ask your administrator to increase PULLDB_SESSION_TTL_HOURS in the server configuration.

Checklist:

  • Username is case-sensitive — verify the exact spelling
  • Your account may have been disabled by an admin
  • An admin may have force-reset your password — check with them for your temporary credentials

If your credentials are correct and the error persists, contact your pullDB administrator to verify your account status.

Symptoms: CLI commands fail with 401 Unauthorized — Authentication required.

Causes:

  • Credentials file missing or corrupted (~/.pulldb/credentials)
  • You have not logged in yet on this machine

Fix:

$ pulldb login

If you are a new user who hasn't registered yet:

$ pulldb register

Symptom: You generated an API key from your profile page but requests using it return 401 or 403.

Cause: API keys require administrator approval before they become active. New keys are in pending state until an admin reviews them.

Fix: Contact your pullDB administrator and ask them to approve your API key. Admins can do this from Admin → API Keys in the web UI.

Service / Container Issues

Symptom: docker ps -a shows the container with status Exited (1).

Check the startup logs:

$ docker logs --tail 100 pulldb

Look for a line beginning with [entrypoint] FATAL: to identify the failure point. Common causes: missing required environment variables, TLS certificate issues, or a bad volume mount.

Symptom: The container is running but the web UI or API is unreachable. pullDB runs its internal services (API, web, worker, MySQL) via supervisord.

Check service status:

$ docker exec pulldb supervisorctl status

If a service shows FATAL or BACKOFF, check its log:

$ docker exec pulldb cat /mnt/data/logs/pulldb/pulldb-api.log
$ docker exec pulldb cat /mnt/data/logs/pulldb/pulldb-web.log
$ docker exec pulldb cat /mnt/data/logs/pulldb/pulldb-worker.log

Restart an individual service:

$ docker exec pulldb supervisorctl restart pulldb-api

Contact your administrator if the service continues to crash — the log output will identify the root cause.

Backup Picker Shows "Loading backups..." Forever

1.3.0 known issue (cached pages): If you upgraded to 1.3.0 but your browser has cached the old restore page, you may still see this symptom. A hard refresh resolves it immediately.

Symptom: After selecting a customer in the restore form, the backup list shows "Loading backups..." indefinitely. No results appear and no error is shown.

Cause: The browser cached the restore page from before the 1.3.0 update. The old page loaded HTMX from an external CDN (unpkg.com), which the server's Content Security Policy now blocks. Without HTMX, the backup search request is never issued and the UI hangs silently.

Fix — hard refresh the page:

  • Windows / Linux: Ctrl+Shift+R
  • macOS: Cmd+Shift+R

A hard refresh bypasses the browser cache and loads the updated page that serves HTMX locally. The backup picker will work normally after one hard refresh.

Verify the fix: Open the browser developer console (F12 → Network tab), reload, and confirm no requests go to unpkg.com. HTMX should load from /static/js/vendor/htmx.min.js.
