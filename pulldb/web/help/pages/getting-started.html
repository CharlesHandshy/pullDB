Getting Started with pullDB

Get from zero to your first restored database in under 5 minutes. This guide covers everything you need to start using pullDB.

~5 minutes
1

Prerequisites

Before you start, make sure you have:

  • Access to a machine with pulldb installed
  • A valid username registered in the system
  • Knowledge of at least one customer ID with available backups
Not sure about your username?
Contact your team admin or check with whoever manages your pullDB installation.
New accounts require admin approval
After running pulldb register, your account and API key start in a pending state. You will see API key pending approval when trying to run commands. Contact your administrator to enable your account before proceeding.
2

Find Your Customer and List Backups

If you don't know the exact customer identifier, search for it first:

Search Customers
$ pulldb search acme
Customers matching 'acme':
acme
acme-west
2 result(s).

Once you know the exact identifier, list its available backups:

List Backups
$ pulldb list acme
Backups for 'acme':
DATE TIME (UTC) SIZE ENV FILENAME
---------- ---------- ------ ---- ----------------------------------
2026-01-02 08:37:36 2.3 GB prod daily_mydumper_acme_2026-01-02...
2026-01-01 08:42:29 2.3 GB prod daily_mydumper_acme_2026-01-01...
2025-12-31 08:40:46 2.2 GB prod daily_mydumper_acme_2025-12-31...
3 backup(s) found.
Restore with: pulldb restore acme

Tip: The highlighted date is the most recent backup. Use date=YYYY-MM-DD to restore a specific version.

3

Submit Your First Restore

Run the restore command with your username and the customer ID:

Submit Restore
$ pulldb restore acme
Job queued successfully!
customer: acme
target: chrlesacme
staging_name: chrlesacme_7f3a2b1c85a1
job_id: 7f3a2b1c-85a1-4da2-9e3f-...
status: queued
user: charles (chrles)
Use 'pulldb status' to monitor progress.
How the database name is built
The target name is your 6-character user code followed by the customer name with all non-letter characters stripped — no separator. For example, user chrles restoring customer acme gets database chrlesacme. A customer like acme-corp would become chrlesacmecorp (hyphen removed). You can override this with target=myname or append a short suffix with suffix=dev.
Job ID is important!
Save your job ID (the first 8 characters like 7f3a2b1c). You'll need it to track progress and access your database.
4

Watch It Work

Follow your restore in real-time with the events command:

Live Events
$ pulldb status --rt
Streaming events for active job 7f3a2b1c...
(Ctrl+C to stop)
[10:42:36] queued: Job submitted
[10:42:56] running: Job started by worker (target: chrlesacme_7f3a2b1c85a1)
[10:42:56] download_started: {"bucket":"pestroutes-rds-backup...}
[10:43:27] download_complete: 2.3 GB downloaded
[10:43:29] extraction_complete: Backup extracted
[10:43:30] restore_started: myloader running
[10:45:30] restore_complete: myloader finished
[10:45:31] post_sql_started: Running post-SQL scripts
[10:45:33] post_sql_complete: 12 scripts executed
[10:45:34] deployed: Database chrlesacme is live
Job 7f3a2b1c finished with status: deployed
What does "deployed" mean?
deployed is the final success state — your database is live and ready to connect to. It will remain available until its retention period expires (typically 7 days) or you release it manually. You can extend the expiry or lock the database to prevent automatic cleanup.

What just happened? pullDB downloaded your backup from S3, extracted it, loaded it into a staging database using myloader, ran post-SQL scripts to anonymize data, and then atomically renamed it to your final target database.

5

Connect to Your Database

Your database is now ready! Connect using your MySQL client:

Connect
$ mysql chrlesacme
Welcome to MySQL...
mysql> SHOW TABLES;
+-------------------+
| Tables_in_chrlesacme |
+-------------------+
| users |
| orders |
| products |
| ...
🎉
You did it!
You've successfully restored your first database with pullDB. The data has been anonymized with post-SQL scripts, making it safe for development.
